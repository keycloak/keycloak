= Security and Standards

== Stronger security for the Identity Brokering API
// https://github.com/keycloak/keycloak/issues/47339

The Identity Brokering API allows applications to retrieve tokens obtained from external identity providers during federated login. Version 2 of this API replaces the legacy V1 with a more secure and standards-compliant design:

* *Client-level authorization* — access to external tokens is controlled per client using dedicated settings (*Allow retrieve external tokens* and an identity provider allow list) instead of assigning broker roles to individual users.
* *Confidential clients only* — public clients are rejected, ensuring that only authenticated clients can retrieve external tokens.
* *OAuth 2.0 compliant* — the endpoint uses `POST` and returns standard JSON responses with `access_token`, `error`, and `error_description` fields.
* *Session-based token storage* — a new *Store token in session* option keeps tokens in the user session for faster access, with automatic cleanup on session expiry. Database storage remains available for persistence across sessions.

V2 is now supported but disabled by default. V1 is deprecated but still enabled by default for backward compatibility. In a future release, V1 will be removed and V2 will become the default.

For more information, see the link:{developerguide_link}#_identity-brokering-apis[Identity Brokering APIs] chapter in the {developerguide_name}.

== Automate user provisioning with the SCIM API (preview)
// https://github.com/keycloak/keycloak/issues/47590

https://datatracker.ietf.org/doc/html/rfc7644[SCIM] (System for Cross-domain Identity Management) is a standard protocol for reading and writing identity resources such as users and groups across multiple systems. It enables organizations to automate user provisioning and deprovisioning using widely available tooling — for example, integrating with identity governance platforms, HR systems, or other identity providers.

{project_name} has SCIM APIs for managing users and groups within a realm. The implementation covers full CRUD and PATCH operations, filtering and pagination, schema extensions including the Enterprise User extension, and schema discovery endpoints.

In this release, the SCIM API is being promoted to a preview feature. In the default profile it is disabled by default; to try it out, enable the `scim-api` feature.

For more details, see the link:{adminguide_link}#_managing_scim[Managing users and groups through SCIM] documentation.

== Real-time security signals to downstream applications (experimental)
// https://github.com/keycloak/keycloak/issues/48901

When a user logs out, changes credentials, or gets disabled in {project_name}, downstream applications typically don't learn about it until the next token refresh — leaving a window where stale sessions remain active. The https://openid.net/specs/openid-sharedsignals-framework-1_0-final.html[OpenID Shared Signals Framework (SSF)] closes this gap by letting identity providers push security events to relying parties in near real time.

{project_name} can now act as an SSF Transmitter, delivering signed https://www.rfc-editor.org/rfc/rfc8417[Security Event Tokens (SETs)] to registered receivers. It supports https://openid.net/specs/openid-caep-1_0-final.html[CAEP 1.0] and https://openid.net/specs/openid-risc-1_0-final.html[RISC 1.0] event profiles with both push (https://www.rfc-editor.org/rfc/rfc8935[RFC 8935]) and poll (https://www.rfc-editor.org/rfc/rfc8936[RFC 8936]) delivery methods. Events are persisted to a durable outbox so that no signal is lost across restarts, and a cluster-aware background drainer handles retries and exponential backoff. Each realm can opt in individually, and streams, subjects, and event types are managed through the admin console and REST API.

To enable SSF, start {project_name} with `--features=ssf`.

For more details, see the link:{securing_apps_base_link}/ssf-support[Shared Signals Framework] guide.

== Standardized authorization decisions with AuthZen (experimental)
// https://github.com/keycloak/keycloak/issues/47629

Applications that need fine-grained authorization typically call {project_name}'s proprietary authorization API, coupling them to its internal model. The https://openid.net/specs/authorization-api-1_0.html[OpenID AuthZEN Authorization API 1.0] defines a vendor-neutral protocol between Policy Decision Points (PDPs) and Policy Enforcement Points (PEPs), enabling applications to request authorization decisions through a standard interface.

{project_name} now implements the AuthZEN Evaluation API as a PDP. Applications send evaluation requests describing a subject, resource, and action, and {project_name} returns a permit or deny decision based on its configured authorization policies. Both single evaluation and batch evaluations endpoints are supported.

To enable AuthZen, start {project_name} with `--features=authzen`.

For more details, see the link:{securing_apps_base_link}/authzen-authorization[AuthZen Authorization] guide.

== Delegate access across services with Token Exchange Delegation (experimental)
// https://github.com/keycloak/keycloak/issues/49954

In multi-service architectures, a backend service sometimes needs to act on behalf of a user at another service — for example, when a gateway delegates work to a downstream API. Standard Token Exchange handles client-to-client token swaps, but verifying that the requester is authorized to impersonate a specific user requires additional checks.

{project_name} introduces token exchange delegation as an experimental feature. It adds a new `delegation` parameterized scope type that validates whether the requesting user is authorized to act on behalf of the target user before the exchange is granted. The delegation scope requires user consent and is automatically reassessed on token refresh, so revoked impersonation rights take effect immediately.

To enable token exchange delegation, start {project_name} with `--features=token-exchange-delegation`.

To try it out, enable the `client-admin-api:v2` feature.

== Fine-grained delegation for organization administration
// https://github.com/keycloak/keycloak/issues/45497

{project_name} now supports delegated organization administration without requiring the broad `manage-realm` role. This is achieved through new dedicated admin roles and Fine-Grained Admin Permissions support for organizations.

New realm admin roles provide coarse-grained delegation:

* `manage-organizations` — grants full read and write access to organizations, including creating, updating, and deleting organizations and their members.
* `view-organizations` — grants read-only access to organizations and their members (also requires `view-users` or Fine-Grained Admin Permissions for user visibility).
* `query-organizations` — grants the ability to search and list organizations without full view access, consistent with the `query-users` / `query-clients` / `query-groups` pattern.

The `manage-realm` role continues to implicitly grant full organization management access for backward compatibility.

For per-organization granularity, organizations are now a first-class resource type in Fine-Grained Admin Permissions. Administrators can create permissions to control which specific organizations a delegated administrator can view or manage — for example, granting access to manage one organization without giving access to all organizations in the realm. When Fine-Grained Admin Permissions is enabled, organization member queries also respect user-level permissions, returning only members the administrator is permitted to view.

== Automatic role inheritance through organization groups
// https://github.com/keycloak/keycloak/issues/47327

Organization groups now support realm and client role assignments. Roles assigned to organization groups are automatically included in `realm_access` and `resource_access` token claims for group members. Additionally, the *Organization Group Membership* protocol mapper (OIDC and SAML) can optionally include these roles within the `organization` claim, organized per organization, by enabling the *Add group role mappings* configuration option.

For more details, see the link:{adminguide_link}#pass:a[_managing_groups_][Managing organization groups] guide.

== Better passkey compatibility with new WebAuthn policy options
// https://github.com/keycloak/keycloak/issues/42513

The WebAuthn Policy and WebAuthn Passwordless Policy provide a new `Discoverable credential` option that follows the current WebAuthn specification.
In addition to requiring or discouraging discoverable credentials (passkeys), it supports the `preferred` value, which was not possible with the previous Yes/No option.
The new option is sent to the browser as `authenticatorSelection.residentKey` when registering WebAuthn credentials, which improves compatibility with passkey providers.
The previous `Require Discoverable Credential` option is now deprecated, and it is planned to be removed in future.

== Type-safe validation for parameterized scopes (experimental)
// https://github.com/keycloak/keycloak/issues/49532

The type is required when creating parameterized scopes with the feature enabled.
