// ------------------------ Breaking changes ------------------------  //
== Breaking changes

Breaking changes are identified as those that might require changes for existing users to their configurations or applications.
In minor or patch releases, {project_name} will only introduce breaking changes to fix bugs.

=== Test Framework representation builders

The test framework provides a number of builders to simplify creating representations. These can be used on their own,
but are also used by `RealmConfig`, `ClientConfig` and `UserConfig` implementations.

Some builders had the suffix `ConfigBuilder`, while others had the shorter syntax `Builder`. They are all now renamed to use
the shorter suffix `Builder` to keep things consistently.

Additionally, some methods where updated for consistency across the builders:

* `RoleBuilder` method `setSingleAttribute` renamed to `attribute`
* `UserBuilder` method `roles` renamed to `realmRoles`
* `RealmBuilder` method `client` and `addClient` replaced with `clients`
* `RealmBuilder` method `user` and `addUser` replaced with `users`
* `RealmBuilder` method `addRole` replaced with `realmRoles`
* `RealmBuilder` method `addClientRole` replaced with `clientRoles`
* `RealmBuilder` method `addAuthenticationFlow` replaced with `authenticationFlows`
* `RealmBuilder` method `addClientScope` replaced with `clientScopes`
* `RealmBuilder` method `identityProvider` renamed to `identityProviders`
* `RealmBuilder` method `identityProviderMapper` renamed to `identityProviderMappers`
* `RealmBuilder` method `requiredAction` renamed to `requiredActions`

=== Identity Provider alias is now immutable

The alias of an Identity Provider can no longer be changed after creation via the Admin REST API. Attempts to update the alias will result in a `400 Bad Request` error.
Previously, although the admin console UI already treated the alias as read-only, the REST API still allowed alias changes.

=== User profile metadata no longer returned on user search

When searching for users using the Admin REST API, the user profile metadata is no longer included for each user in the search results.

This change was made to improve performance of user search operations, as fetching user profile metadata for each user can be resource-intensive.
The user profile metadata can still be retrieved when fetching a specific user by their identifier.

=== Organization member query returns brief user representation by default

When querying organization members using the Admin REST API (`GET /admin/realms/\{realm}/organizations/\{id}/members`), a brief user representation is now returned by default.
To retrieve the full user representation, pass `?briefRepresentation=false` as a query parameter.
The single-member endpoint (`GET /admin/realms/\{realm}/organizations/\{id}/members/\{member-id}`) continues to return the full representation.

=== X509 client authentication requires CA Subject DN

Since this release, the X509 client authentication, that allows a client authenticate presenting a certificate in the mutual TLS (mTLS) handshake, requires to configure a new option called **Certificate Authority subject DN** to identify the trusted anchor CA for the client certificate. As the {project_name}'s trust-store (or mTLS store) is shared for all the realms, the new option allows to identify the correct CA of the certificate and avoids interferences with other unrelated CAs.

The new option is compulsory in the administration console when configuring X509 credentials for the client. For the moment, previous configurations without the CA subject DN still work to maintain backwards compatibility. In the next major version, the validation will be present at the server side, making any update, creation or import invalid if the new configuration option is not present. Please, if you use the X509 authenticator for clients ensure you add this new attribute to avoid issues in the future.

Besides, the option **Allow regex pattern comparison** is deprecated. This option allows to configure a regexp instead of the exact DN of the client certificate. It is now deprecated and it will be removed in a future version. The certificate for the client should be unambiguously identified.

For the link:https://www.keycloak.org/securing-apps/client-registration[client registration service], there is a new link:{adminguide_link}#_client_policies[client policy executor] named `tls-client-auth-ca-subject-dn` which can be used to assign a default CA subject DN. The executor can also enforce the configured CA DN to reject unwanted CAs.

In general, all this new configuration requires {project_name} to always revalidate the trusted anchors for the client certificate. This feature can be problematic in certain configurations (for example if a proxy is configured as a TLS termination and the certificates are passed to {project_name} using headers). The trust-store or mTLS store in {project_name} should always be configured correctly to validate the client certificate chain. Please see link:https://www.keycloak.org/server/reverseproxy[Using a reverse proxy] and link:https://www.keycloak.org/server/mutual-tls[Configuring trusted certificates for mTLS], for more information if you are using a TLS termination proxy and X509 client authentication.

// ------------------------ Notable changes ------------------------ //
== Notable changes

Notable changes may include internal behavior changes that prevent common misconfigurations, bugs that are fixed, or changes to simplify running {project_name}.
It also lists significant changes to internal APIs.

=== Organization invitation filters use exact matching

The `email`, `firstName`, and `lastName` query parameters on the organization invitations list endpoint (`GET /admin/realms/{realm}/orgs/{orgId}/invitations`) now use case-insensitive exact matching instead of substring matching.

If you rely on partial matching for these fields, use the `search` query parameter instead, which performs case-insensitive substring matching across email, firstName, and lastName fields.

=== Dynamic Scopes feature renamed to Parameterized Scopes

The experimental feature `dynamic-scopes` has been renamed to `parameterized-scopes`. If you enabled this feature using `--features=dynamic-scopes`, update your configuration to use `--features=parameterized-scopes`.

The corresponding Java model methods in `ClientScopeModel` (`isDynamicScope()`, `setIsDynamicScope()`, `getDynamicScopeRegexp()`) and constants (`IS_DYNAMIC_SCOPE`, `DYNAMIC_SCOPE_REGEXP`) have been deprecated in favor of their `parameterized` counterparts. The deprecated methods and constants will be removed in a future major release. The underlying database attribute names (`is.dynamic.scope`, `dynamic.scope.regexp`) are automatically migrated to `is.parameterized.scope` and `parameterized.scope.regexp` during upgrade.

=== Asynchronous commit on PostgreSQL databases

Transactions that update only ephemeral tables like persisted user sessions, client sessions, login failures, or events will use PostgreSQL's async commit feature.
Logouts will still force a synchronous commit.

You can opt out of this feature by setting `+--spi-connections-jpa--quarkus--async-commit=false+`

=== Parameterized scopes now require a parameter type

When the `parameterized-scopes` feature is enabled, creating or updating a parameterized scope via the Admin REST API now requires a `parameterized.scope.type` attribute. Available types are `string`, `integer`, `boolean`, `username`, and `custom`. The `custom` type additionally requires a `parameterized.scope.regexp` attribute with a valid regular expression. Requests using the old regexp-only format (without a type) are rejected.

During upgrade, existing parameterized scopes are automatically migrated to preserve their current behavior:

* Scopes with the default `<scope-name>:*` regexp are set to the `string` type and the regexp is removed.
* Scopes with a custom regexp are set to the `custom` type and the scope name prefix is stripped from the regexp.
* Scopes with no regexp are set to the `string` type.

No manual action is required — the migration ensures existing scopes continue to work as before.

=== Verify email required before credentials setup during user self-registration

When user self-registration is enabled for the realm together with *Verify Email*, then users will not see password by default
on the registration screen. After registration of user's profile, user would be required to verify email and then he can setup password
(or eventually other credentials like OTP or Passkey) as a subsequent step. It is possible to disable and stick to the old behavior by
enable switch *Always set password on register form* on the realm registration flow on the *Password validation* authenticator. Note
that this option is deprecated and exists mostly for the backwards compatibility. It may be removed in the future.

For the details, see the link:{adminguide_link}[{adminguide_name}].

=== Socket timeouts and query timeouts for database connections

For all database connections, {project_name} now sets a socket read timeout matching the respective transaction timeout to prevent long hanging threads. For PostgreSQL databases, it additionally sets a query timeout.

=== Configure TOTP and Update password required actions moved after Verify Email

In relation to the previous point, the required actions *Configure OTP* and *Update password* are moved in the order of required actions after *Verify Email*
required action. This is done automatically for new realm or during server update. If you prefer the old order, you can manually move required actions per your preference.

=== Shutdown timeout extended

Previous versions waited for one second for HTTP requests to finish during the shutdown of a node.

The current version extends the timeout to ten seconds by default. In setups that use distributed caches, it now also waits for caches to rebalance before shutting down, preventing a possible data loss.
When there are no ongoing HTTP requests, and caches are already rebalanced, the shutdown will finish immediately as before. With that, we expect most customers to not observe a change in the behavior of {project_name}.

As before, for any rolling update or scaling up or down, we require to do this one node at a time for all clustered setups of {project_name}.

Use the `shutdown-timeout` option to adjust the timeout to your environment. Restore the original behavior by setting it to `1s`.

=== Extensions must not restart a Keycloak transaction

This change might affect custom {project_name} extensions.

The implicit transaction handling in {project_name} previously assumed that a transaction is started once per session, which is usually request-scoped. If it was started a second time, this might have led to an inconsistent state.

Starting with this version, this is now enforced and a session's transaction can only be started once. This prevents difficult-to-track errors and lost commits. It also ensures that at the start of the transaction, all nested transactions are started if they have not been started earlier.

A transaction is started automatically before the REST endpoint code is invoked, even if the session was created earlier in a filter.

As before, transactions can be nested, which will pause the outer transaction once the nested transaction starts, and will resume the outer transaction once the nested transaction ends.

=== Organization group representations now include role mappings

Organization group representations returned by the Admin REST API now include `realmRoles` and `clientRoles` fields. Previously, these fields were `null` for organization groups. They are now empty lists/maps by default, and populated when roles are assigned to the group. If your code checks for `null` to detect organization groups or conditionally processes role mappings, update it to handle the new behavior.

=== Realm search now matches by display name

When searching for realms in the admin console, the search now also matches against the realm's display name in addition to the realm name.

As part of this change, the `displayName` attribute has been promoted from a realm attribute to a dedicated column on the realm entity.

Before migrating, check the current value of the `displayName` attribute for your realms. If the current value of `displayName` attribute
exceeds 255 characters, it will be truncated during migration.

=== LDAP binary attribute decoding is now configurable

The User Attribute LDAP mapper now includes a *Decode Binary Attribute As* configuration option that controls how binary LDAP attributes are decoded. The available values are:

* *auto* (default for new mappers) — decodes as UUID when the LDAP attribute matches the configured UUID LDAP attribute (e.g. `objectGUID` in Active Directory), base64 otherwise.
* *base64* — always returns a base64-encoded string.
* *uuid* — always decodes the value as a UUID/GUID string. If the binary value is not exactly 16 bytes (the size of a UUID), the decoder falls back to base64 encoding.

This change reverts a regression introduced in 26.6.1 where the `objectGUID` binary attribute format was silently changed from base64 to UUID string for user attribute mappers. Existing binary attribute mappers are automatically migrated to use *base64* decoding to preserve their current behavior. If you were relying on the UUID string format introduced in 26.6.x, explicitly set the decoder to *uuid* after upgrading.

The Group LDAP mapper also includes a new *Decode UUID Attribute to UUID Format* toggle. When enabled, the UUID LDAP attribute (e.g. `objectGUID` in Active Directory) listed in "Mapped Group Attributes" is decoded to UUID string format instead of base64. New group mappers default to enabled (UUID format). Existing group mappers are automatically migrated to disabled (base64) to preserve their current behavior.

=== DPoP not supported for implicit and hybrid flows

DPoP (Demonstrating Proof of Possession) is no longer supported for implicit and hybrid flows. If a client has the *Require DPoP bound tokens* option enabled, authorization requests using implicit or hybrid response types will be rejected with an error.

This change improves security by preventing the use of DPoP with flows that issue access tokens in the front channel, where the sender-constrained token model cannot be properly enforced.

=== Token Exchange with Sender-constrained tokens allowed for scope changes

Standard Token Exchange now accepts sender-constrained tokens when a client is requesting an exchange of their own token.
This change allows token audience changes and up/down scoping and is applicable to both DPoP-bound tokens and X.509 certificate-bound tokens (mTLS).

If you attempt to exchange a sender-constrained token using a different client, the request will be rejected with an `invalid_request` error.
For more details, see the link:{securing_apps_token_exchange_link}[Token Exchange] guide.

=== HAProxy certificate lookup provider
The `HaProxySslClientCertificateLookup` implementation has deprecated support for configuring `--spi-x509cert-lookup--haproxy--ssl-cert-chain-prefix`
as HAProxy does not provide a built-in mechanism to forward individual intermediate certificates as dedicated HTTP headers.
Consequently, this solution does not work when more than one intermediate certificate is used. Existing deployments of
`HaProxySslClientCertificateLookup` will continue to work with `ssl-cert-chain-prefix` as before, but a deprecation warning
will now be logged.

Users of `ssl-cert-chain-prefix` should instead make use of `--spi-x509cert-lookup--haproxy--ssl-cert-chain`.
See the link:{client_certificate_lookup_link}[Reverse Proxy guide] for more details.

=== Async Custom REST Endpoints should not be used

Async custom REST endpoints using `AsyncResponse` or returning an async type such as `CompletableFuture` or `CompletionState` should not be used. If you are using them, be aware that the `KeycloakSession` used to initial the request will be immediately closed - it will not remain open for the entire lifespan of the request - and that you are fully responsible for transaction management in the async work.

=== Validation of client_session_host parameter
The `client_session_host` parameter is now validated against the client's registered nodes and management URL.
This parameter is sent by clients during token refresh and is used when the `${application.session.host}` placeholder is present in a client's backchannel logout URL configuration.
It was primarily used by legacy adapters that have been removed.
This improves security by ensuring that only trusted, pre-registered hosts can be used in backchannel logout URLs.

// ------------------------ Deprecated features ------------------------ //
== Deprecated features

The following sections provide details on deprecated features.

=== Identity Brokering API V1 is deprecated

The Identity Brokering API V1 is now deprecated but remains enabled by default for backward compatibility. See the link:{releasenotes_link}[Release Notes] for more details.

=== Switch Bearer-only on OpenID Connect clients is deprecated

The switch **Bearer only** on OIDC clients is deprecated now and will be removed in some future version.

The switch was possible to be used to specify that client can act just as service, which is not able to authenticate by any grant and obtain tokens issued to itself. However bearer-only client can
be already simulated by creating OIDC client without any grants supported.

The switch is already unavailable in the admin console client settings, but it was still possible to use it for example by directly updating client through admin REST API. In the
future, this will not be possible anymore.

=== Option Allow regex pattern comparison is deprecated for the X509 client authenticator

The option **Allow regex pattern comparison** is deprecated for the X509 client authenticator. This option allows to configure a Java regular expression instead of the exact DN for the client certificate subject DN. As previously commented in the breaking changes section, this option is now deprecated and it will be removed in a future version. The certificate for the client should be unambiguously identified because of security reasons.

=== WebAuthn policy option `Require Discoverable Credential` deprecated in favor of `Discoverable Credential`

The WebAuthn policy option `Require Discoverable Credential` (`requireResidentKey`) is deprecated and planned to be removed in the future.
A new option `Discoverable Credential` (`residentKey`) is available in the WebAuthn Policy and WebAuthn Passwordless Policy, following the current https://www.w3.org/TR/webauthn/#dom-authenticatorselectioncriteria-residentkey[WebAuthn specification].
Unlike the deprecated Yes/No option, it supports the values `required`, `preferred` and `discouraged`, and is sent to the browser as `authenticatorSelection.residentKey` during the registration of WebAuthn credentials.

The deprecated option keeps working and is only used when `Discoverable Credential` is left as `not specified`.
A warning is logged when the deprecated option is set to a concrete value.
The corresponding realm representation attributes `webAuthnPolicyRequireResidentKey` and `webAuthnPolicyPasswordlessRequireResidentKey` are deprecated in favor of `webAuthnPolicyResidentKey` and `webAuthnPolicyPasswordlessResidentKey`.

// ------------------------ Removed features ------------------------ //
== Removed features

The following features have been removed from this release.

=== Batching option for persistent user sessions removed

The internal SPI options to enable writing updates to persistent sessions in batches have been removed.
Batching has been disabled in previous releases by default, and its performance proved to be lower than with batching disabled, leading to its removal in this release.

The options `spi-user-sessions--infinispan--use-batches` and `spi-user-sessions--infinispan--max-batch-size` have been removed without replacement.
