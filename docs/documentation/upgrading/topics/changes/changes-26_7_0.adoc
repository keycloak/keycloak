== Breaking changes

Breaking changes are identified as those that might require changes for existing users to their configurations or applications.
In minor or patch releases, {project_name} will only introduce breaking changes to fix bugs.

=== Test Framework representation builders

The test framework provides a number of builders to simplify creating representations. These can be used on their own,
but are also used by `RealmConfig`, `ClientConfig` and `UserConfig` implementations.

Some builders had the suffix `ConfigBuilder`, while others had the shorter syntax `Builder`. They are all now renamed to use
the shorter suffix `Builder` to keep things consistently.

Additionally, some methods where updated for consistency across the builders:

* `RoleBuilder` method `setSingleAttribute` renamed to `attribute`
* `UserBuilder` method `roles` renamed to `realmRoles`
* `RealmBuilder` method `client` and `addClient` replaced with `clients`
* `RealmBuilder` method `user` and `addUser` replaced with `users`
* `RealmBuilder` method `addRole` replaced with `realmRoles`
* `RealmBuilder` method `addClientRole` replaced with `clientRoles`
* `RealmBuilder` method `addAuthenticationFlow` replaced with `authenticationFlows`
* `RealmBuilder` method `addClientScope` replaced with `clientScopes`
* `RealmBuilder` method `identityProvider` renamed to `identityProviders`
* `RealmBuilder` method `identityProviderMapper` renamed to `identityProviderMappers`
* `RealmBuilder` method `requiredAction` renamed to `requiredActions`

=== Identity Provider alias is now immutable

The alias of an Identity Provider can no longer be changed after creation via the Admin REST API. Attempts to update the alias will result in a `400 Bad Request` error.
Previously, although the admin console UI already treated the alias as read-only, the REST API still allowed alias changes.

== Notable changes

Notable changes may include internal behavior changes that prevent common misconfigurations, bugs that are fixed, or changes to simplify running {project_name}.
It also lists significant changes to internal APIs.

=== Verify email required before credentials setup during user self-registration

When user self-registration is enabled for the realm together with *Verify Email*, then users will not see password by default
on the registration screen. After registration of user's profile, user would be required to verify email and then he can setup password
(or eventually other credentials like OTP or Passkey) as a subsequent step. It is possible to disable and stick to the old behavior by
enable switch *Always set password on register form* on the realm registration flow on the *Password validation* authenticator. Note
that this option is deprecated and exists mostly for the backwards compatibility. It may be removed in the future.

For the details, see the link:{adminguide_link}[{adminguide_name}].

=== Configure TOTP and Update password required actions moved after Verify Email

In relation to the previous point, the required actions *Configure OTP* and *Update password* are moved in the order of required actions after *Verify Email*
required action. This is done automatically for new realm or during server update. If you prefer the old order, you can manually move required actions per your preference.

=== Shutdown timeout extended

Previous versions waited for one second for HTTP requests to finish during the shutdown of a node.

The current version extends the timeout to ten seconds by default. In setups that use distributed caches, it now also waits for caches to rebalance before shutting down, preventing a possible data loss.
When there are no ongoing HTTP requests, and caches are already rebalanced, the shutdown will finish immediately as before. With that, we expect most customers to not observe a change in the behavior of {project_name}.

As before, for any rolling update or scaling up or down, we require to do this one node at a time for all clustered setups of {project_name}.

Use the `shutdown-timeout` option to adjust the timeout to your environment. Restore the original behavior by setting it to `1s`.

=== Extensions must not restart a Keycloak transaction

This change might affect custom {project_name} extensions.

The implicit transaction handling in {project_name} previously assumed that a transaction is started once per session, which is usually request-scoped. If it was started a second time, this might have led to an inconsistent state.

Starting with this version, this is now enforced and a session's transaction can only be started once. This prevents difficult-to-track errors and lost commits. It also ensures that at the start of the transaction, all nested transactions are started if they have not been started earlier.

A transaction is started automatically before the REST endpoint code is invoked, even if the session was created earlier in a filter.

As before, transactions can be nested, which will pause the outer transaction once the nested transaction starts, and will resume the outer transaction once the nested transaction ends.

=== Realm search now matches by display name

When searching for realms in the admin console, the search now also matches against the realm's display name in addition to the realm name.

As part of this change, the `displayName` attribute has been promoted from a realm attribute to a dedicated column on the realm entity.

Before migrating, check the current value of the `displayName` attribute for your realms. If the current value of `displayName` attribute
exceeds 255 characters, it will be truncated during migration.

=== DPoP not supported for implicit and hybrid flows

DPoP (Demonstrating Proof of Possession) is no longer supported for implicit and hybrid flows. If a client has the *Require DPoP bound tokens* option enabled, authorization requests using implicit or hybrid response types will be rejected with an error.

This change improves security by preventing the use of DPoP with flows that issue access tokens in the front channel, where the sender-constrained token model cannot be properly enforced.

=== Token Exchange with Sender-constrained tokens allowed for scope changes

Standard Token Exchange now accepts sender-constrained tokens when a client is requesting an exchange of their own token.
This change allows token audience changes and up/down scoping and is applicable to both DPoP-bound tokens and X.509 certificate-bound tokens (mTLS).

If you attempt to exchange a sender-constrained token using a different client, the request will be rejected with an `invalid_request` error.
For more details, see the link:{securing_apps_token_exchange_link}[Token Exchange] guide.
