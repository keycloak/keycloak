=== Unspecific redirect URIs

Make your registered redirect URIs as specific as feasible. Registering vague redirect URIs for xref:con-oidc-auth-flows_{context}[Authorization Code Flows] can allow malicious clients to impersonate another client with broader access. Impersonation can happen if two clients live under the same domain, for example.

You can use the secure redirect uris enforcer executor for your realm. The result makes sure that client administrators are able to register only clients with specific redirect-uris matching various requirements such as requiring that a URL cannot have wildcards in the context path, restricting wildcard subdomains to the first DNS label, or limiting redirects to specified permitted domains. See <<_client_policies, Client Policies>> for details about how to configure client policies with a specific executor.

The `allow-wildcard-subdomain` setting on the secure redirect uris enforcer executor controls whether client registration and update requests may include wildcard-host redirect URIs when that executor applies. This setting is disabled by default.

Wildcard subdomains are not recommended for production use; see RFC 9700 section 4.1. Use them only when necessary and only for domains where every matching subdomain is trusted. A wildcard such as `https://*.example.com/callback` matches a single host label such as `https://preview.example.com/callback`, but not `https://foo.preview.example.com/callback`.

A common use case is dynamic preview environments with hostnames like `https://app-<id>.example.com`, where `<id>` is assigned at deployment time and is not known when the client is configured. A wildcard-host redirect URI such as `https://app-*.example.com/callback` covers all preview deployments while remaining scoped to the `example.com` domain.

When a client uses `+` for Web Origins, origins derived from wildcard-host redirect URIs remain wildcard origins. For example, `https://*.example.com/callback` derives the origin `https://*.example.com`.
