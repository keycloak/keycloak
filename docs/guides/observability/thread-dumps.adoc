== What is a thread dump?

A JVM thread dump is a snapshot of the execution state of active threads within a Java Virtual Machine at a specific point in time.

Thread dumps are useful for diagnosing issues such as thread contention, deadlocks, CPU spikes, or stuck IO operations.

== How to capture a JVM thread dump

There are several ways how to capture a thread dump. This guide used `jcmd` as it provides the most straightforward command line experience.

The `jcmd` tool sends diagnostic commands to a running JVM. There are two commands relevant for capturing thread dumps: `Thread.print` and `Thread.dump_to_file`.

* `jcmd <PID> Thread.print -l` Prints all threads with stack traces to the standard output. The `-l` option enables printing additional information about locks.
* `jcmd <PID> Thread.dump_to_file threaddump.txt` Writes thread stack traces to a file in a plain text format.
* `jcmd <PID> Thread.dump_to_file -format=json threaddump.json` Writes thread stack traces to a file in a JSON format.

There are some differences between the information collected using `Thread.print` and `Thread.dump_to_file` which are important when investigating issues related to virtual threads. There are also differences in the output of `Thread.dump_to_file` between JDK 21 and JDK 25.

|Aspect | `Thread.print` | `Thread.dump_to_file` on JDK 21 | `Thread.dump_to_file` on JDK 25

| Output target

| Output format
| Plain text
| Plain text or JSON
| Plain text or JSON

| Locks/monitors
| Detailed
| *Omitted* due to performance issues
| Detailed (restored)

| Virtual thread visibility
| *Only if mounted*
| Comprehensive
| Comprehensive

| Performance overhead
| High. Forces JVM into global safepoint. Not recommended under heavy load.
| Lower. Designed to work with high number of virtual threads and to avoid global safepoint.
| Lower. Designed to work with high number of virtual threads and to avoid global safepoint.

|===

Depending on the information needed and the JDK version used, it might be necessary to use both `Thread.print` and `Thread.dump_to_file` to create a comprehensive picture.

=== Finding PID of the {project_name} process

There are several ways to find a PID of the {project_name} server process on the system.

Running the `jcmd` command without any parameters will list out Java processes on the system and their PIDs.

[source]
----
242551 io.quarkus.bootstrap.runner.QuarkusEntryPoint start ...
268778 jdk.jcmd/sun.tools.jcmd.JCmd
----

=== Collecting multiple samples

Because a thread dump is a snapshot of the JVM state at a particular point in time, it is recommended to capture multiple thread dumps, for example 3-5 samples spaced 5-10 seconds apart.
