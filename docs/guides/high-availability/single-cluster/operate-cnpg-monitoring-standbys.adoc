== When to use this procedure

In a CloudNativePG cluster deployed in high availability mode, standby instances are critical for both data durability and failover readiness.
Monitoring standby health helps detect replication issues early and ensures a safe promotion candidate is available when needed.

== Prerequisites

* A CloudNativePG cluster deployed according to steps described in the <@links.ha id="single-cluster-deploy-cnpg" /> {section}.
* The `kubectl` command-line utility.
* The `kubectl cnpg` plugin.
Please follow the https://cloudnative-pg.io/docs/${properties["cnpg.major-minor-version"]}/kubectl-plugin[CloudNativePG documentation] for installation steps.
* Prometheus and Grafana installed on the {kubernetes} cluster.
Basic installation steps can be found in the https://cloudnative-pg.io/docs/${properties["cnpg.major-minor-version"]}/quickstart[Quickstart section of the CloudNativePG documentation].

== Procedure

. Review the status of the CloudNativePG cluster using the `kubectl cnpg status` command.
+
.Command:
[source,bash]
----
kubectl cnpg status -n cnpg-keycloak cnpg-keycloak
----
+
.Output:
[source,bash]
----

<1> The cluster status should read `Cluster in healthy state`.
Any other value indicates a problem.
<2> This section shows the status of the cluster's standby instances and their replication health.
<3> General status of individual instances and their roles in the cluster.

. Verify standby health in the _Streaming Replication status_ table.
+
The following fields help determine whether standbys are healthy and replication is working:
+
[%autowidth]
|===
| Field | Expected value | What it means

| `State`
| `streaming`
| The standby is connected to the primary and actively receiving WAL data.

| `Sync State`
| `quorum` or `sync`
| Indicates synchronous replication is active for this standby. In a quorum-based synchronous setup, at least one standby should be in `quorum`.

| `Write Lag`, `Flush Lag`, `Replay Lag`
| `00:00:00`
| Replication lag metrics. A non-zero value that grows over time indicates the standby is falling behind.

| `Replication Slot`
| `active`
| Confirms the replication slot is in use and the standby is consuming WAL.
|===

. Check that there is at least one safe standby candidate.
+
In the `Sync State` column, standbys with values `quorum` or `sync` are safe promotion candidates.
Standbys in `potential` or `async` are replicating asynchronously at that moment.
+
.Sync State
[%autowidth]
|===
| Sync State | Replication | Safe to promote

| `quorum` | Quorum Synchronous | Safe
| `sync` | Synchronous | Safe
| `potential` | Asynchronous ^1^ | Unsafe
| `async` | Asynchronous | Unsafe
|===
+
.Table footnotes:
^1^ May be promoted to a synchronous standby if the current synchronous standby fails.
+
[IMPORTANT]
====
If all standbys are `potential` or `async`, the cluster is not currently providing synchronous replication guarantees.
Investigate standby health and replication before performing maintenance operations.
====

. Enable collection of metrics from all cluster instances, including standbys, by creating a `PodMonitor` resource:
+
.Command:
[source,yaml,indent=0]
----
kubectl -n cnpg-keycloak apply -f - <<EOF
apiVersion: monitoring.coreos.com/v1
kind: PodMonitor
metadata:
  name: cnpg-keycloak-pod-monitor
spec:
  selector:
    matchLabels:
      cnpg.io/cluster: cnpg-keycloak <1>
  podMetricsEndpoints:
  - port: metrics
EOF
----
<1> Name of the CloudNativePG cluster to be monitored.
The selector targets all pods in the cluster, including standbys.

. Import the CloudNativePG Grafana dashboard to visualise standby metrics:
+
.Command:
[source,bash]
----
curl -L -O https://raw.githubusercontent.com/cloudnative-pg/grafana-dashboards/main/charts/cluster/grafana-dashboard.json

kubectl create configmap cnpg-dashboard \
  --from-file=cnpg-dashboard.json=grafana-dashboard.json \
  -n monitoring

kubectl label configmap cnpg-dashboard \
  grafana_dashboard=1 \
  -n monitoring
----

. To access the Grafana dashboard, forward the Grafana service port and retrieve the admin password:
+
.Command:
[source,bash]
----
kubectl port-forward -n monitoring svc/prometheus-grafana 3001:80

# In a separate terminal, retrieve the admin password:
kubectl get secret --namespace monitoring prometheus-grafana \
  -o jsonpath="{.data.admin-password}" | base64 --decode; echo
----
+
Open `http://localhost:3001` in a browser and log in with username `admin` and the password retrieved above.
When adding Prometheus as a data source, use `DS_PROMETHEUS` as the data source variable name and `prometheus` as the Prometheus service name.

. Once Prometheus is scraping metrics, use the following key metrics in Grafana to observe standby health:
+
[%autowidth]
|===
| Metric | Description

| `cnpg_pg_replication_lag`
| Replication lag in seconds per standby instance. A value near `0` is healthy.

| `cnpg_pg_replication_in_recovery`
| Returns `1` if the instance is a standby (in recovery mode), `0` for the primary.

| `cnpg_pg_replication_is_wal_receiver_up`
| Returns `1` if the WAL receiver is running on the standby. A value of `0` indicates a broken replication stream.

| `cnpg_pg_stat_replication_write_lag`
| Time elapsed between WAL flushed on the primary and received by the standby.

| `cnpg_pg_stat_replication_replay_lag`
| Time elapsed between WAL flushed on the primary and replayed on the standby.
|===

== What healthy looks like

A healthy standby setup typically shows:

* Cluster status is `Cluster in healthy state`.
* All standby instances show `State: streaming`.
* `Write Lag`, `Flush Lag`, and `Replay Lag` stay at `00:00:00` (or near-zero) and do not trend upward.
* At least one standby has `Sync State: quorum` (for quorum-based synchronous replication as described in the <@links.ha id="single-cluster-deploy-cnpg" /> {section}).
* `cnpg_pg_replication_is_wal_receiver_up` is `1` for all standbys in Prometheus.

== Signs of an unhealthy standby

The following are indicators that a standby requires attention:

* The cluster `Status` is not `Cluster in healthy state`.
* A standby `State` is not `streaming`.
* Any of `Write Lag`, `Flush Lag`, or `Replay Lag` is non-zero and increasing over time.
* No standby is in `quorum` or `sync` state when synchronous replication is expected.
* A standby is missing from the _Streaming Replication status_ table.
* `cnpg_pg_replication_is_wal_receiver_up` is `0` for any standby in Prometheus.

If one or more standby instances show these symptoms, investigate using the following commands:

* Verify that the standby pods are running:
+
.Command:
[source,bash]
----
kubectl -n cnpg-keycloak get pods -L role
----

* Check recent events in the namespace for scheduling, image pull, storage, or networking problems:
+
.Command:
[source,bash]
----
kubectl -n cnpg-keycloak get events --sort-by=.lastTimestamp | tail -n 30
----

* Inspect the CloudNativePG cluster resource for conditions and related messages:
+
.Command:
[source,bash]
----
kubectl -n cnpg-keycloak describe cluster cnpg-keycloak
----

For possible troubleshooting scenarios refer to the https://cloudnative-pg.io/docs/${properties["cnpg.major-minor-version"]}/troubleshooting[CloudNativePG documentation].
